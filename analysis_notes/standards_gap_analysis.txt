  STANDARDS GAP ANALYSIS — SimuLTE Mode4 Implementation
  Date: 2026-03-10
  Standards: IEEE 1609.2-2022, SAE J3161/1 (JUL 2024), SAE J2735-2024

EXECUTIVE SUMMARY
-----------------
The SimuLTE Mode4 implementation is structurally well-aligned with all three
standards. The SPDU, Certificate, and BSM message definitions closely mirror
the IEEE 1609.2 ASN.1 layout. Most J3161 radio parameters are now compliant.
BSM Part I contains all 19 J2735 fields.

The remaining gaps fall into three categories:
  1. Behavioral checks not implemented (PSID validation, replay protection, etc.)
  2. PKI/trust model simplifications (self-signed certs, no CA chain)
  3. One major radio-layer gap (SPS + One-Shot not implemented)

These are standard simulation simplifications that do not affect the primary
research contribution: measuring computational and bandwidth overhead of PQC
signature algorithms (Falcon-512, Dilithium-2) versus ECDSA P-256.

SEVERITY LEVELS:
  [CRITICAL]  - Fundamentally affects security model or radio behavior
  [HIGH]      - Missing mandatory behavior required by the standard
  [MEDIUM]    - Missing commonly-used feature
  [LOW]       - Simplification acceptable for simulation

================================================================================
SECTION 1: IEEE 1609.2-2022 GAPS
================================================================================

--- CRITICAL (1) ---

GAP-1609-C1: NO TRUST MODEL / CERTIFICATE CHAIN VALIDATION
  Reference: IEEE 1609.2-2022 Section 5.2.6.2.1
  Status: NOT IMPLEMENTED
  Detail:
    All certificates are self-signed (issuerType=1, self). No Root CA trust
    anchors, no PCA/ICA hierarchy, no chain traversal. Receiver uses the
    certificate's own public key with no issuer verification. Certificate
    validity is set to INT64_MAX (never expires).
  Impact:
    The security layer measures crypto computational and bandwidth overhead
    correctly, but the trust model is non-existent. This is standard practice
    in V2X simulation (Veins, Artery, etc.) and does not affect the radio
    performance metrics under study.

--- HIGH (3) ---

GAP-1609-H1: PSID CONSISTENCY CHECK NOT IMPLEMENTED
  Reference: IEEE 1609.2-2022 Section 5.2.6.3.2
  Status: Fields present, check NOT implemented
  Detail:
    SPDU has psid field, Certificate has appPermPsid field. Receiver never
    compares them. A certificate authorized for ICA (PSID 0x40) could sign
    BSMs (PSID 0x20) and the receiver would accept it.

GAP-1609-H2: SERVICE SPECIFIC PERMISSIONS (SSP) ABSENT
  Reference: IEEE 1609.2-2022 Section 6.4.28
  Status: NOT IMPLEMENTED
  Detail:
    No SSP field in Certificate.msg. SSP provides fine-grained control over
    which optional BSM fields a certificate is authorized to assert.

GAP-1609-H3: CONTENT TYPE CHOICE DISCRIMINATOR MISSING
  Reference: IEEE 1609.2-2022 Section 6.3.2
  Status: MISSING
  Detail:
    Ieee1609Dot2Content is a CHOICE between unsecuredData, signedData,
    encryptedData, etc. No contentType discriminator field exists in SPDU.msg;
    all packets are implicitly treated as SignedData.

--- MEDIUM (6) ---

GAP-1609-M1: HASH ALGORITHM FIELD MISSING
  Reference: IEEE 1609.2-2022 Section 6.3.5
  Detail: SignedData requires hashId (sha256/sha384/sm3). Currently implicit
    from algorithm choice.

GAP-1609-M2: SIGNATURE TYPE DISCRIMINATOR MISSING
  Reference: IEEE 1609.2-2022 Section 6.3.13
  Detail: Signature is raw bytes; algorithm determined from cert algoName
    string. No type tag in the SPDU Signature field itself.
  Note: PQC algorithms (Falcon-512, Dilithium-2) are NOT part of the current
    1609.2 Signature CHOICE type. This is expected and is a key research
    contribution — evaluating PQC candidates as future replacements.

GAP-1609-M3: GENERATION TIME VALIDATION NOT PERFORMED
  Reference: IEEE 1609.2-2022 Sections 5.2.6.2.3, 5.2.7.2.2
  Detail: generationTime is set at send but never checked against certificate
    validity period, freshness window, or current time at receiver.

GAP-1609-M4: REPLAY PROTECTION NOT IMPLEMENTED
  Reference: IEEE 1609.2-2022 Section 5.2.7.2
  Detail: No duplicate detection. Same signed message could be replayed.
    Acceptable for benign simulation environment.

GAP-1609-M5: IMPLICIT CERTIFICATES (ECQV) NOT SUPPORTED
  Reference: IEEE 1609.2-2022 Section 6.4.4
  Detail: Only explicit certificates (certType=0). Implicit certs use EC key
    reconstruction for smaller wire size.

GAP-1609-M6: PSEUDONYM / LINKAGE CERTIFICATES NOT IMPLEMENTED
  Reference: IEEE 1609.2-2022 Section 6.4.9
  Detail: Each node uses a single permanent certificate. No privacy rotation,
    no linkageData, no pseudonym certificate pool.

--- LOW (3) ---

GAP-1609-L1: GEOGRAPHIC REGION IN CERTIFICATES
  Reference: IEEE 1609.2-2022 Section 6.4.17
  Detail: No GeographicRegion field in Certificate.msg. Cannot perform
    geographic consistency checks.

GAP-1609-L2: EXPIRY TIME IN HEADERINFO
  Reference: IEEE 1609.2-2022 Section 6.3.9
  Detail: expiryTime not present in SPDU.msg. Optional for BSMs.

GAP-1609-L3: COER ENCODING
  Reference: IEEE 1609.2-2022 Section 6.1
  Detail: Uses OMNeT++ native serialization instead of Canonical OER. This is
    standard practice in all OMNeT++/Veins simulation work. Wire byte lengths
    are computed analytically for accurate overhead comparison.

SECTION 2: SAE J3161/1 (JUL 2024) GAPS
================================================================================

--- CRITICAL (1) ---

GAP-J3161-C1: SPS + ONE-SHOT NOT IMPLEMENTED
  Reference: SAE J3161 Section 7.3.3, Figure 15
  Status: NOT IMPLEMENTED
  Detail:
    "LTE-V2X shall use one-shot transmission along with SPS flow to avoid
    repetitive packet collision." (Section 7.3.3)
    "SPS + one-shot shall be applied to all SPS flows generated by LTE-V2X
    radio (e.g., OBU and RSU)."
    Requires OneShotCtr alongside ResourceReselectionCtr (Table 4):
      - OneShotCtrMin=2, OneShotCtrMax=6 (for RRI >= 100ms)
      - OneShotCtrMin=4, OneShotCtrMax=12 (for RRI = 50ms)
    Current simulation uses SPS-only scheduling in LteMacVUeMode4.cc.
  Impact:
    Without One-Shot, the simulation cannot mitigate repetitive packet
    collisions caused by half-duplex resource reservation conflicts.
  Fix: Implement the SPS + One-Shot scheme per Figure 15 flowchart in
    LteMacVUeMode4.cc with OneShotCtr alongside ResourceReselectionCtr.

--- MEDIUM (1) ---

GAP-J3161-M1: MCS 8, 9, 10 EXCLUSION — LOGIC ONLY, NOT FORMALLY VERIFIED
  Reference: SAE J3161 Section 8.6
  Status: PARTIALLY IMPLEMENTED
  Detail:
    MCS range is now correctly set to minMCS=5, maxMCS=11. Code comments in
    LteMacVUeMode4.cc (lines 1068, 1228) reference the MCS 8/9/10 exclusion
    rule. However, there is no formal unit test or assertion proving these MCS
    values are never selected during resource allocation. The exclusion depends
    on the MCS selection logic skipping these values correctly.

SECTION 3: SAE J2735-2024 GAPS
================================================================================

--- LOW (2) ---

GAP-J2735-L1: BSM PART II OPTIONAL EXTENSIONS NOT IMPLEMENTED
  Reference: SAE J2735 Part II
  Status: NOT IMPLEMENTED
  Detail: Only Part I (BSMcoreData) is implemented with all 19 fields. Part II
    optional extensions (PathHistory, PathPrediction, etc.) are not present.
  Impact: Part II is optional and not relevant to the security overhead study.

GAP-J2735-L2: UPER ENCODING NOT USED
  Reference: SAE J2735 ASN.1 encoding rules
  Status: NOT IMPLEMENTED
  Detail: BSM uses flat C++ struct fields, not ASN.1 UPER encoding. This is
    standard practice in OMNeT++ simulation. Wire byte lengths are computed
    analytically.

SECTION 4: DELIBERATE SIMULATION-SPECIFIC DEVIATIONS
================================================================================

These are intentional deviations for simulation purposes, not gaps:

1. msgId (int32_t) in BSM
   - NOT in J2735 (which only has msgCnt). Used for PDR tracking in simulation.
   - Adds 4 bytes to BSM wire length (accounted for in byte calculations).

2. subjectId, algoName (strings) in Certificate
   - NOT in 1609.2 wire format. Used for simulation debugging and algorithm ID.
   - NOT counted in wire byte length.

3. Lat/Lon encoding uses millimeters (coord * 1000), not 1/10 micro-degrees
   - J2735 defines lat/lon as 1/10 micro-degree for WGS84 GPS coordinates.
   - OMNeT++/SUMO uses Cartesian meters, not GPS. Using * 1e7 overflows int32
     for meter-scale coordinates. Using * 1000 gives mm precision within int32.
   - Decode path consistently uses / 1000.0 on all receivers.

4. ECDSA public key stored as OpenSSL DER SubjectPublicKeyInfo internally
   - ECDSA P-256: 91 bytes DER vs 65 bytes raw uncompressed point.
   - Wire byte length SUBTRACTS the 26-byte DER overhead for ECDSA.
   - PQC algorithms (Falcon, Dilithium) have no DER wrapper.

5. Self-signed certificates (see GAP-1609-C1)
   - No CA hierarchy. issuerType=1 (self) for all nodes.
   - Standard practice in V2X simulation research.

SECTION 7: RECOMMENDED PAPER FRAMING
================================================================================

"Our simulation implements the IEEE 1609.2-2022 Signed SPDU security model with
explicit certificates, digest-based certificate caching (HashedId8), and all 19
SAE J2735 BSMcoreData Part I fields. The LTE-V2X Mode 4 radio layer follows SAE
J3161/1 deployment profiles including CBR-based congestion control, compliant MCS
range (5-11), and SPS resource allocation with probResourceKeep=0.8.

Management-plane features (PKI hierarchy, CRL, SCMS, pseudonym rotation) are
simplified as they do not affect the radio-layer performance metrics under study.
Certificate attachment follows SAE J2945/1 policy: full certificate every Nth
packet, HashedId8 digest otherwise.

We evaluate PQC candidates (Falcon-512, Dilithium-2) as drop-in replacements for
the ECDSA P-256 signature algorithm defined in the IEEE 1609.2 Signature CHOICE
type, measuring computational overhead, packet size impact, and PDR under
realistic channel conditions."
