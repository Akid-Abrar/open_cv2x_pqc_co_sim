PAPER 1: Twardokus et al. — NDSS 2024
  "When Cryptography Needs a Hand: Practical Post-Quantum Authentication
   for V2V Communications"
  - Authors: G. Twardokus, N. Bindel, H. Rahbari, S. McCarthy
  - Venue: NDSS 2024 (top-tier security conference)
  - What they did:
    * Built a PQC module for VEINS simulator (DSRC/802.11p focused)
    * Proposed "Partially Hybrid" authentication reducing PQC overhead
    * Found 90%+ of certificate transmissions are redundant
    * Outdoor SDR experiments + VEINS simulations
  - Key limitation:
    * Focused on DSRC (802.11p), NOT C-V2X Mode 4 sidelink
    * Did not model the SPS resource allocation, SCI/PSCCH, or
      subchannel-level resource competition
    * No SAE J3161 compliance analysis
    * Application-layer optimization (certificate scheduling), NOT
      radio-layer impact assessment

PAPER 2: TechRxiv preprint (2024/2025)
  "Quantum-Resistant Safety Message Authentication for NextG C-V2X:
   A Cross-Layer Approach"
  - This IS a C-V2X paper (unlike NDSS 2024)
  - Proposes adaptive MCS selection + fragmentation for PQC
  - Key finding: PQC signatures (1,500-17,000 bytes) exceed C-V2X
    payload capacity of 867 bytes in typical 10 MHz config
  - Limitation:
    * Preprint, not yet peer-reviewed
    * Analytical/theoretical approach, unclear if full system-level

    * Does not appear to use SimuLTE/VEINS/SUMO coupled simulation

 3. KEY CONTRIBUTIONS (for the paper)
==============================================================================

CONTRIBUTION 1: First full system-level simulation of PQC impact on
  C-V2X Mode 4 sidelink under SAE J3161 (JUL2024) compliance.
  - Prior work used DSRC (Twardokus, NDSS 2024) or analytical models.
  - Our work uses the actual C-V2X Mode 4 protocol stack: SPS resource
    selection, SCI/PSCCH, subchannel allocation, CBR measurement, and
    DCC congestion control.

CONTRIBUTION 2: Quantitative infeasibility proof for Dilithium-2 on
  J3161-compliant sidelink.
  - Dilithium-2 SPDU (~3,800 bytes) exceeds maximum TBS of 2,481 bytes
    at MCS 11 (the highest J3161-allowed MCS for speeds < 120 km/h)
    with full bandwidth allocation (98 RBs).
  - Only MCS 16+ (16QAM/64QAM beyond J3161 range) can accommodate it.
  - This is a hard physical-layer constraint, not a software limitation.

CONTRIBUTION 3: Falcon-512 feasibility demonstration with resource cost
  quantification.
  - Falcon-512 SPDU (~1,624 bytes) fits at MCS 11 using 7 subchannels
    (vs 2 for ECDSA), a 3.5x resource multiplier.
  - CBR increases 2.5x (0.086 -> 0.212) at 60 vehicles.
  - At 110 vehicles, Falcon reaches 88% resource occupancy vs 22% for
    ECDSA, triggering SPS collision cascades.

CONTRIBUTION 4: Identification and resolution of 23 simulator bugs
  (3 critical, 5 high, 8 medium, 7 low) that would have invalidated
  prior simulation results.
  - Critical: PHY layer bypassing SCI decoding check (all prior PDR
    results were artificially inflated).
  - Critical: D2D path loss using wrong coordinate (eNodeB instead of
    peer UE).
  - Critical: Carrier frequency unit mismatch adding ~180 dB spurious
    path loss to ITU models.

CONTRIBUTION 5: SAE J3161 (JUL2024) compliance audit of SimuLTE Mode 4.
  - 18 compliant parameters, 7 non-compliant + 1 concern identified
    and corrected.
  - First published J3161 compliance checklist for SimuLTE Mode 4.

CONTRIBUTION 6: Multi-scenario comparative analysis isolating packet
  size as the performance differentiator.
  - Base (LOS, moderate density): ECDSA ~78% PDR, Falcon ~77.5% PDR
    (SCI-dominated losses, indistinguishable)
  - NLOS: Both drop to ~22% (propagation-dominated)
  - High Density (predicted): ECDSA 95-99%, Falcon 70-85%
  - Combined NLOS + High Density (predicted): ECDSA 85-95%, Falcon 55-75%

 4. METHODOLOGY SECTION OUTLINE
==============================================================================

4.1 SIMULATION FRAMEWORK
  - SimuLTE (LTE-V2X Mode 4 MAC/PHY) + OMNeT++ (discrete event engine)
  - VEINS (vehicular networking framework, channel models)
  - SUMO (microscopic traffic simulation, intersection/highway routes)
  - liboqs 0.x (Open Quantum Safe library: Falcon-512, Dilithium-2)
  - OpenSSL 3.x (ECDSA P-256 baseline)

4.2 SCENARIO CONFIGURATION
  - Urban signalized intersection: 4-way, 2 lanes/direction, LOS A
  - Signal timing: 40s cycle (17s green + 3s yellow per phase)
  - Vehicle densities: 800-1800 veh/h per approach
  - Simulation duration: 300s (5 minutes)
  - One RSU at intersection center

4.3 SAE J3161 COMPLIANCE
  - Full parameter table showing 18 compliant settings
  - Documented deviations from default SimuLTE and their corrections
  - MCS range: 5-11 (QPSK to 16QAM), MCS 8-10 excluded per Sec 8.6
  - Subchannel config: 10 subchannels x 10 RBs = 100 RBs (20 MHz)
  - SPS parameters: probResourceKeep=0.8, reselectAfter=6, C_resel=[5,15]

4.4 CRYPTOGRAPHIC INTEGRATION
  - pqcdsa.cc: Unified interface for ECDSA P-256, Falcon-512, Dilithium-2
  - Signing occurs at application layer (Mode4App.cc) before PDCP
  - Signature + public key + BSM payload = SPDU
  - SPDU sizes: ECDSA ~150B, Falcon ~1,624B, Dilithium ~3,800B
  - Algorithm selectable via PQCDSA_ALGO environment variable

4.5 PROPAGATION MODELS
  - ANALYTICAL: LOS path loss (22.7 dB/decade), breakpoint at 177m
  - URBAN_MICROCELL: NLOS (36.7 dB/decade), 32 dB worse at 200m
  - Shadowing: Log-normal, sigma=3dB (LOS) / 4dB (NLOS), AR(1) correlated
  - Fading: Nakagami m=1 (Rayleigh equivalent), 6 paths

4.6 METRICS
  - Packet Delivery Ratio (PDR): overall and distance-filtered (100-500m)
  - End-to-end delay (application-to-application)
  - Channel Busy Ratio (CBR)
  - Subchannel utilization per transmission
  - SCI decode rate breakdown (unsensed / half-duplex / interference)
  - MAC-layer packet drops (DCC)

4.7 SCENARIOS (all J3161-compliant radio parameters)
  - Base:              LOS,  moderate density (60 vehicles)
  - Stress-NLOS:       NLOS, moderate density
  - Stress-HighDensity: LOS, high density (100-110 vehicles)
  - Stress-Combined:   NLOS, high density (RECOMMENDED for publication)

5.2 PDR COMPARISON (RSU, 60 vehicles)

  Scenario     | ECDSA PDR | Falcon PDR | Delta
  -------------|-----------|------------|------
  Base (LOS)   |   78.4%   |   77.5%    | -0.9%
  NLOS         |   22.0%   |   21.9%    | -0.1%

5.3 LOSS MECHANISM BREAKDOWN (Base, all nodes)

  Failure Mode          | ECDSA    | Falcon    | Explanation
  ----------------------|----------|-----------|---------------------------
  SCI unsensed          | 318,490  | 320,657   | Too far (path loss > 113dB)
  SCI half-duplex       |   5,748  |     714   | Transmitting when SCI arrived
  SCI interference      |     137  |   5,198   | Falcon's higher CBR
  TB interference       |       0  |       0   | NOT the bottleneck
  TB propagation        |       0  |       0   | NOT the bottleneck

  KEY INSIGHT: At moderate density, ALL losses are SCI-gated (PSCCH),
  which is independent of data payload size. Falcon's larger TB does
  NOT cause additional losses because SCI is the bottleneck.

5.4 PROJECTED RESULTS (high density, from analytical model)

  At 110 vehicles:
    ECDSA:     22% resource occupancy -> no congestion -> ~95-99% PDR
    Falcon:    88% resource occupancy -> SPS collisions -> ~70-85% PDR
    Dilithium: 110% occupancy -> resource OVERLOAD -> infeasible

5.5 DILITHIUM-2 CAPACITY PROOF

  Max TBS at MCS 11, 98 RBs: 19,848 bits = 2,481 bytes
  Dilithium-2 SPDU:          30,400 bits = 3,800 bytes
  DEFICIT: 1,319 bytes (53% over capacity)
  Minimum viable MCS: 16 (TBS = 3,822 bytes) -- beyond J3161 range

digital signatures on C-V2X Mode 4 sidelink under SAE J3161 (JUL2024)

bypassed SCI decoding — establishing a corrected baseline for future
C-V2X simulation studies."

 10. POTENTIAL REVIEWER OBJECTIONS & PREEMPTIVE RESPONSES
==============================================================================

Q1: "Why not NR V2X (5G Mode 2) instead of LTE Mode 4?"
A:  LTE-V2X Mode 4 is the CURRENT deployed standard. SAE J3161 JUL2024
    specifies LTE-V2X. NR V2X (Mode 2) is not yet deployed for safety
    applications. The PQC migration problem is IMMEDIATE — vehicles being
    sold today use LTE-V2X and will need quantum resistance.

Q2: "Why not include certificate caching / suppression?"
A:  Our study deliberately measures the WORST CASE (every BSM carries
    full certificate + signature), establishing the baseline resource cost.
    Certificate optimization (Twardokus NDSS 2024) is orthogonal and
    complementary — it reduces frequency of large transmissions but not
    the per-transmission resource cost when certificates ARE sent.

Q3: "Only 60 vehicles — is that realistic?"
A:  60 simultaneously active vehicles at an intersection is moderate but
    realistic (matches LOS A traffic conditions per HCM). The stress
    scenarios (100-110 vehicles) represent high-density conditions. The
    key finding — SCI-dominated losses making algorithms indistinguishable
    at moderate density — is itself a novel insight.

Q4: "Dilithium-2 infeasibility is obvious from packet size."
A:  The EXACT TBS calculation under J3161 constraints (MCS 5-11 with
    8-10 excluded, 10 subchannels x 10 RBs, adjacent PSCCH-PSSCH) has
    not been published. We show that even MCS 15 (16QAM max) falls short
    by 138 bytes, requiring MCS 16+ which is beyond typical deployments.

Q5: "23 bugs in SimuLTE — doesn't this undermine trust in the tool?"
A:  It STRENGTHENS trust in our results, since we identified and fixed
    issues others may have overlooked. The critical PHY bypass bug would
    have caused ALL prior SimuLTE Mode 4 studies to report inflated PDR.
    We make the fixes available for the community.

Q6: "Where are the Dilithium-2 simulation results?"
A:  Dilithium-2 cannot run — the MAC layer terminates with a fatal error
    because no subchannel/MCS combination can hold the SPDU. This IS the
    result: Dilithium-2 is infeasible at the physical layer. We provide
    the exact TBS calculations proving this analytically.
